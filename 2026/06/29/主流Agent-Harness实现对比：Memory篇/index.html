全篇主线

1. 从 RAG 到 LLM Wiki

传统 RAG 的基本流程

传统 RAG 适合大量静态资料问答，例如企业知识库、论文库、客服手册。

传统 RAG 的局限

  • 每次查询都像从零开始查资料
  • LLM 没有持续积累
  • 复杂问题需要临时拼接多个片段
  • chunk 容易碎片化，缺少整体结构
  • 相似度高不等于当前任务真正有用

例如一个问题需要综合 5 篇文档，传统 RAG 每次都要重新检索、拼接、总结；系统本身不会自然沉淀出“已经整理好的知识”。

LLM Wiki 的思路

LLM Wiki 不是只在查询时检索原始文档，而是让 LLM 持续维护一个长期 Wiki。

它的重点是：把查过、总结过、综合过的内容沉淀成结构化 Markdown 页面，之后可以继续维护和复用。

对比项 传统 RAG LLM Wiki
核心动作 检索片段 建设知识库
信息形态 chunk Markdown 页面
使用方式 每次重新检索 先读已有整理，再补充更新
长期积累 弱 强
维护方式 重新索引为主 持续写入、更新、交叉引用

2. LLM Wiki 的结构与运行方式

三层结构

原始资源层

原始资源是事实来源，通常只读不改。LLM 可以读取它们，但不应该直接改写原文。

Wiki 层

Wiki 是 LLM 生成和维护的 Markdown 文件集合。它负责：

  • 创建页面
  • 更新页面
  • 合并重复内容
  • 维护交叉引用
  • 保持概念一致性
  • 把好的回答归档为长期页面

视图 / 规则层

类似 CLAUDE.md、AGENTS.md，它告诉 Agent：

  • Wiki 的目录结构是什么
  • 新资源如何导入
  • 回答问题时先读哪些文件
  • 如何更新索引和日志
  • 如何处理冲突、过期和引用

这类文件让 LLM 从普通聊天模型变成一个有规范的 Wiki 维护者。

3. 索引与日志

index.md：内容索引

index.md 面向内容，通常记录：

  • 页面链接
  • 一句话摘要
  • 分类
  • 日期
  • 来源数量
  • 可选 metadata

作用是让 LLM 先通过索引判断该读哪些页面，而不是盲目全文搜索。

适合规模：中等规模 Wiki，例如百级来源、数百个页面。这种方式可以避免一开始就上复杂的 embedding / 向量数据库架构。

log.md：时间日志

log.md 面向时间，记录 Wiki 发生过什么：

  • 什么时候导入了什么资源
  • 什么时候回答了什么问题
  • 什么时候更新了哪些页面
  • 什么时候做了检查

4. Memory 的读与写

Memory 写什么

Memory 不应该保存所有聊天记录，而应该保存长期有用的信息。

适合写入：

  • 用户偏好
  • 项目背景
  • 架构约定
  • 工具使用经验
  • 常见错误和避坑经验
  • 已确认的技术决策
  • 需要长期遵守的规则

不适合写入：

  • 临时闲聊
  • 一次性任务细节
  • 已经过期的信息
  • 没有复用价值的过程信息
  • 未确认的猜测
  • 密钥、隐私、敏感信息

Memory 怎么写

写入 Memory 时要判断：

好的 Memory 不只是事实，还应该说明为什么重要、以后怎么用。

Memory 读什么

读 Memory 的目标不是“多读”，而是读当前任务真正需要的少量高价值记忆。

读取时要检查：

  • 是否和当前任务相关
  • 是否适用于当前项目 / 文件 / 模块
  • 是否过期
  • 是否和当前代码冲突
  • 是否有更新、更可靠的信息
  • 是否应该先核查再使用

5. 为什么 Coding Agent 不常用传统向量 RAG

原因 1：没有大量同类内容召回需求

通用 Agent Harness 通常不是在海量同类文档中找答案，而是需要记住当前项目约定、用户偏好、历史决策、工具链细节和常见错误。

原因 2：单个 workspace 的 Memory 规模有限

一个项目里真正值得长期记住的内容通常没有大到必须依赖向量数据库。文本文件 + 索引文件已经能覆盖很多场景。

原因 3：Memory 比普通 RAG 文档更珍贵

普通 RAG 召回错一段文档，可能只是回答质量下降；Memory 召回错了，可能直接导致 Agent 做错事。

例如：

  • 使用过期架构约定
  • 误用旧的用户偏好
  • 根据错误历史决策修改代码
  • 忽略当前代码中的新事实

所以 Coding Agent 的 Memory 更强调：

向量召回的合理位置

向量召回不是没用，而是不适合包打天下。它适合大量文本的语义检索，但 Agent Memory 还需要额外处理冲突、过期、权限、来源、适用范围和当前代码核查。

6. 两种 Memory 读写范式

范式一：Tool + Prompt 主动读写

做法：通过 Prompt 引导 LLM，在需要时调用工具。

优点：

  • 实现轻量
  • 接入简单
  • 适合交互式 Agent
  • 可以快速响应当前对话中的记忆需求

缺点：

  • 模型不一定知道什么该记
  • 容易写入低质量 Memory
  • 难以稳定处理复杂整合、冲突和过期
  • 占用 tool call 轮次，拉长响应时间

范式二：异步离线 Memory 整理

做法：不在用户交互时完成所有 Memory 整理，而是在后台周期性处理。

优点：

  • 更适合深度整理
  • 不阻塞当前用户请求
  • 有利于长期积累
  • 可以降低主流程延迟

缺点：

  • 成本更高
  • 速度较慢
  • 周期性执行，不够实时
  • 不方便快速更新刚刚产生的重要记忆

Dream 的定位

Claude Code 的 Dream 可以理解为异步离线 Memory 整理。

8. 其他 Agent Harness 方案

OpenCode / Pi / Kimi Code

这类工具的 Memory 生态里有一些第三方插件方案，常见方向包括：

  • 类 Claude Code 的文本记忆方案
  • 插件式 Memory 管理
  • 部分方案尝试向量召回

例如：

  • opencode-supermemory
  • opencode-mem

其中有些方案实现了向量召回，但仍然要面对 Memory 场景下的准确性、过期、冲突和延迟问题。

OpenClaw

OpenClaw 在底层配置上支持不同 Memory 存储方案，也支持多种 LLM provider。

方案 特点
RAG 式 Memory 底层仍然走传统检索增强思路
LLM Wiki 式 Memory 更接近长期知识页面维护，默认不一定开启

这说明目前业界还没有统一的 Agent Memory 标准，不同 Harness 仍在不同方向上试探。

9. 复习总结

最重要的三个判断

RAG、LLM Wiki、Agent Memory 的关系

概念 重点 类比
RAG 查资料辅助回答 搜索引擎
LLM Wiki 持续维护长期知识 知识手册
Agent Memory 为 Agent 行动提供历史上下文 工作记忆 + 长期笔记

两种 Memory 实现范式

范式 优点 缺点
Tool + Prompt 主动读写 轻量、实时、易接入 难整合、占 tool call、延迟增加
异步离线整理 整合能力强、适合长期积累 成本高、不实时、更新慢

一句话背诵

Agent Memory 的核心是：少记垃圾、勤做整理、召回准确、使用前核查。
